# formally-bmad-implementation-contracts

## Overview

This workflow is the implementation companion to `bmad-dev` for projects that already have approved contract-bearing scaffolds. Act as a formal-implementation governor: implementation may fill in bodies, but it must preserve approved signatures and contracts, and it must add the body-level proof artifacts needed for later verification.

This workflow does not replace `bmad-dev`. It constrains and enriches implementation work that begins from scaffolds generated by `formally-bmad-dsl-contract-stubs`.

Never end the session silently. Before any pause, stop, or handoff, explicitly tell the user what implementation surfaces were completed, which formal additions were preserved or added, what remains blocked, and the exact next workflow.

## Preconditions

Do not begin implementation unless all of the following are true:

- `formally-bmad-dsl-contracts` completed for the story;
- `formally-bmad-dsl-contract-stubs` completed for the story;
- the scaffold was explicitly approved by the user;
- the target verification/tooling profile was fixed;
- the implementation target corresponds to the approved scaffold rather than an ad hoc source layout.

If any precondition is missing, stop and report the gating failure.

## Implementation Rules

When working with `bmad-dev` or implementing code directly from the approved scaffold:

- preserve the generated signature, module, and contract surface unless the user explicitly authorizes a repair cycle;
- keep the chosen notation family consistent with the approved scaffold;
- add only the business logic and the additional body-level proof artifacts needed for full verification.

### Mandatory Body-Level Additions

Where required by the implementation shape, add:

- loop invariants;
- loop variants or termination measures;
- helper lemmas, spec helpers, or proof-oriented utility definitions;
- local assertions and local assumptions that make the proof obligations explicit and auditable.

Do not omit these merely because the code compiles or tests pass.

## Escalation Rules

If implementation reveals that the approved scaffold or upstream contracts are structurally wrong:

- stop implementation work at the mismatch;
- report which contract, scaffold element, or upstream obligation is no longer tenable;
- route back to the repair stage rather than silently rewriting the formal surface.

## Handoff

End with:

- implemented code scope;
- preserved contract surfaces;
- body-level proof artifacts added;
- any mismatches requiring repair;
- mandatory next workflow when implementation exists: `formally-bmad-code-verification`.
